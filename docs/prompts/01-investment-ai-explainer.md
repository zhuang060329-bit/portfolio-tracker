# 01. 投資組合 AI 解釋器

## 定位

升級你現有的投資追蹤網站。目標不是再做一個行情看板，而是讓 AI 解釋 QQQM、BTC、現金、匯率與配置偏離造成的資產變化。

## 第二階段：後端先行規格

需求確認後，先產出後端規格，不做 UI。至少包含以下內容。

### 規格前置規則

- 先取得現有 Supabase schema（資料表、欄位、RLS 現況），由我提供或由 Claude Code 讀 repo 後匯出。
- 規格以 migration delta 形式輸出：標明沿用、修改、新增的表；禁止忽略既有結構做全新設計。

### 建議資料表

- `users`：使用者基本資料與權限。
- `accounts`：券商、交易所、錢包或手動帳戶。
- `assets`：標的基本資料，例如 QQQM、BTC、TWD、USD。
- `transactions`：交易紀錄，含買入、賣出、轉入、轉出、費用、配息。
- `holdings_snapshots`：每日或每週持倉快照。
- `market_prices`：標的價格歷史。
- `fx_rates`：匯率歷史。
- `allocation_targets`：使用者配置目標，例如 QQQM 60%、BTC 40%。
- `portfolio_metrics`：計算後的市值、本金、損益、偏離、回撤等快取。
- `ai_reports`：每週 AI 報告內容、生成時間、使用資料範圍。
- `report_jobs`：定時報告任務狀態、錯誤、重試紀錄。
- `notification_channels`：Telegram、Email 或其他推送設定。

### 核心計算邏輯

請定義公式與邊界情況：

- 成本基礎計算方式：平均成本、FIFO，或 MVP 只用投入本金。
- 市值計算：數量 × 市價 × 匯率。
- 損益計算：市值 - 投入本金 - 費用。
- 配置比例：各資產市值 / 總市值。
- 配置偏離：目前比例 - 目標比例。
- 損益歸因約束：價格變動、匯率、投入本金、費用各因子加總必須等於總變化；無法歸因的差額列為顯式殘差項，不可隱藏或攤入其他因子。
- 下次投入參考：在不賣出的前提下，下一筆資金如何讓配置接近目標。
- 匯率影響：原幣報酬與台幣報酬拆分。
- 缺價處理：最近可用價格、標記資料缺口，不可默默用 0。
- 報告資料範圍：本週起迄、上週對照、資料截止時間。

### API 契約

請設計 endpoint、request、response、錯誤碼：

- `GET /api/portfolio/summary`
- `GET /api/portfolio/allocation`
- `GET /api/portfolio/performance?range=7d|30d|ytd|all`
- `GET /api/transactions`
- `POST /api/transactions`
- `PATCH /api/transactions/:id`
- `DELETE /api/transactions/:id`
- `GET /api/market-prices`
- `POST /api/reports/generate`
- `GET /api/reports/latest`
- `GET /api/reports/:id`
- `POST /api/notifications/test`

每個 endpoint 必須包含：
- 認證與授權規則。
- 輸入驗證。
- 成功 response 範例。
- 錯誤 response 範例。
- 是否可快取。

### 安全要求

- 所有使用者資料必須依 user id 隔離。
- Supabase RLS 必須在第一版就設計。
- API key 僅能放 server side。
- AI 報告生成前要限制可送入模型的資料欄位，避免把不必要的私人資訊送出。
- 金額與配置計算後端為準，前端只顯示。
- 所有 webhook / cron endpoint 必須驗證 secret。
